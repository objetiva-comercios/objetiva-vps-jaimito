**Alpine.js x-data "dashboard" component (copiar patron de RESEARCH.md Pattern 2, con estos detalles):**

Propiedades:
- `metrics: []` — array de MetricResponse del API
- `loading: true` — estado inicial de carga
- `error: false` — estado de error de fetch
- `lastUpdated: ''` — timestamp HH:MM:SS local
- `openMetric: null` — nombre de la metrica con grafico expandido (solo una a la vez per D-07)
- `_pollingTimer: null` — referencia al setInterval
- `_chartInstance: null` — referencia a la instancia uPlot actual

Metodos:
- `init()` — llama fetchMetrics(), inicia setInterval cada 30000ms (DASH-04)
- `destroy()` — clearInterval + chart.destroy()
- `fetchMetrics()` — fetch('/api/v1/metrics'), actualiza metrics[], lastUpdated con toLocaleTimeString('es-AR', {hour12: false}), maneja error state
- `toggleMetric(name, thresholds)` — accordion logic per D-05/D-07: si openMetric === name, colapsa (null + destroy chart); si no, expande nueva (destroy anterior, fetch readings?since=24h per D-06, $nextTick -> renderChart)
- `renderChart(name, readings, thresholds)` — instancia uPlot en `#chart-{name}`. Guard: si readings.length < 2, no instanciar (Pitfall 2). Opciones de uPlot: width=el.clientWidth, height=250, serie con stroke #3b82f6 width 2, ejes con stroke #64748b, ticks/grid #334155. Plugin de threshold lines si thresholds existe (Pattern 3 de RESEARCH.md).
- `sparklinePoints(metric)` — genera string de puntos para SVG polyline. Usa metric.readings (ultimos 20 puntos). Normaliza valores al rango [2, 22] en viewBox 0 0 80 24. Si menos de 2 puntos retorna '' (per RESEARCH.md sparklinePoints).

**NOTA SOBRE SPARKLINES:** El endpoint GET /api/v1/metrics NO retorna array de readings recientes. Solo retorna last_value. Para sparklines, hay dos opciones:
1. No mostrar sparklines hasta que se expanda el grafico (simplifica mucho).
2. Hacer un fetch adicional por metrica para obtener readings recientes.

Dado que la UI-SPEC requiere sparklines en la tabla (DASH-02), la solucion es: al hacer fetchMetrics(), TAMBIEN hacer un fetch de readings corto para cada metrica para tener datos del sparkline. Pero esto genera N+1 requests. Alternativa mas simple: usar los datos ya disponibles (last_value solo) y mostrar un sparkline plano o vacio. O guardar los valores en el cliente conforme llegan via polling (acumulando en un array local).

**Solucion recomendada para sparklines:** Mantener un Map en el Alpine data que acumula valores por metrica conforme llegan via polling cada 30s. Despues de varias iteraciones se forma un sparkline. Inicialmente (antes de acumular datos) el sparkline esta vacio. Agregar propiedad `sparkData: {}` (objeto nombre->array de valores recientes, max 20). En fetchMetrics(), por cada metrica, pushear last_value a sparkData[name] y truncar a 20 elementos.
